# Ubuntu - Guide de Configuration et Sécurisation

::callout{icon="i-heroicons-information-circle"}
📌 **TL;DR**: Guide complet pour sécuriser un serveur Ubuntu dès le provisionnement : configuration système, gestion des utilisateurs, durcissement SSH et bonnes pratiques de sécurité.
::

## Vue d'ensemble

Ubuntu est l'une des distributions Linux les plus populaires pour les serveurs, offrant stabilité, sécurité et une vaste communauté de support. Cette documentation couvre les configurations essentielles pour déployer et sécuriser un serveur Ubuntu en production.

Que vous déployiez sur un VPS (Virtual Private Server), un serveur dédié, ou dans le cloud, les premières heures après le provisionnement sont critiques pour établir une base sécurisée. Une configuration appropriée du système, une gestion rigoureuse des utilisateurs et un durcissement SSH sont les fondations d'une infrastructure robuste.

Ce guide s'adresse aux développeurs fullstack et aux administrateurs système qui souhaitent suivre les meilleures pratiques de sécurité dès le départ.

## Prérequis

- Accès root ou sudo à un serveur Ubuntu (20.04 LTS, 22.04 LTS, ou versions ultérieures)
- Connexion SSH fonctionnelle
- Connaissances de base en ligne de commande Linux
- Client SSH installé localement (OpenSSH, PuTTY, ou terminal macOS/Linux)

::callout{icon="i-heroicons-cube"}
📦 **Versions recommandées** : Ubuntu 22.04 LTS ou 24.04 LTS pour un support long terme et les dernières fonctionnalités de sécurité.
::

## Configuration du Système

### Configuration du Clavier

La disposition du clavier est souvent incorrecte sur les serveurs fraîchement provisionnés. Ubuntu utilise `dpkg-reconfigure` pour reconfigurer les paramètres système interactivement.

::steps

#### Reconfigurer la disposition du clavier

```bash [keyboard-setup.sh]
# Lancer l'assistant de configuration du clavier
sudo dpkg-reconfigure keyboard-configuration
```

::callout{icon="i-heroicons-information-circle"}
Sélectionnez votre modèle de clavier (généralement "Generic 105-key PC"), puis votre disposition (ex: "French - French (AZERTY)").
::

#### Redémarrer le service

```bash [keyboard-restart.sh]
# Appliquer les changements sans redémarrage complet
sudo service keyboard-setup restart
```

#### Configurer les locales système

```bash [locales-setup.sh]
# Définir la langue et les formats régionaux
sudo dpkg-reconfigure locales
```

::callout{icon="i-heroicons-light-bulb" color="blue"}
#summary
Recommandation Locales

#content
Sélectionnez `fr_FR.UTF-8` pour le français avec support Unicode complet, puis définissez-le comme locale par défaut.
::

::

### Mise à Jour du Système

La première action après le provisionnement doit toujours être une mise à jour complète du système pour patcher les vulnérabilités connues.

:::code-group

```bash [update.sh]
# Mettre à jour la liste des paquets disponibles
sudo apt update

# Mettre à niveau tous les paquets installés
sudo apt upgrade -y

# Nettoyer les paquets obsolètes
sudo apt autoremove -y
```

```bash [full-upgrade.sh]
# Mise à jour incluant les nouveaux noyaux et dépendances système
sudo apt update
sudo apt full-upgrade -y
sudo apt autoremove -y
sudo apt autoclean

# Redémarrer si un nouveau noyau a été installé
sudo reboot
```

:::

::callout{icon="i-heroicons-exclamation-triangle" color="amber"}
⚠️ **Attention** : Un redémarrage (`reboot`) peut interrompre temporairement les services. Planifiez cette opération pendant une fenêtre de maintenance.
::

### Gestion des Mots de Passe

```bash [passwd.sh]
# Modifier le mot de passe du compte actuel
passwd

# Modifier le mot de passe d'un autre utilisateur (root requis)
sudo passwd nom_utilisateur
```

::callout{icon="i-heroicons-shield-check" color="amber"}
#summary
Sécurité des Mots de Passe

#content
Utilisez des mots de passe forts (16+ caractères, mixant majuscules, minuscules, chiffres et symboles). Envisagez l'utilisation d'un gestionnaire de mots de passe comme Bitwarden ou 1Password.
::

## Gestion des Utilisateurs

### Principe du Moindre Privilège

L'utilisation directe du compte `root` est une pratique dangereuse. Créez plutôt un utilisateur avec des privilèges sudo limités.

::steps

#### Créer un nouvel utilisateur

```bash [create-user.sh]
# Créer un utilisateur avec répertoire home
sudo adduser nom_utilisateur

# Alternative avec options complètes
sudo useradd -m -s /bin/bash -G sudo nom_utilisateur
```

::callout{icon="i-heroicons-information-circle"}
💡 **Différence** : `adduser` est interactif et plus convivial, `useradd` est plus direct et scriptable.
::

#### Accorder les privilèges sudo

```bash [grant-sudo.sh]
# Ajouter l'utilisateur au groupe sudo
sudo usermod -aG sudo nom_utilisateur

# Vérifier l'appartenance aux groupes
groups nom_utilisateur
```

#### Tester la connexion

```bash [test-user.sh]
# Se connecter avec le nouvel utilisateur
su - nom_utilisateur

# Tester les privilèges sudo
sudo apt update
```

#### Désactiver le login root SSH (recommandé)

Une fois l'utilisateur sudo créé et testé :

```bash [disable-root-ssh.sh]
# Éditer la configuration SSH
sudo nano /etc/ssh/sshd_config

# Ajouter ou modifier cette ligne
PermitRootLogin no

# Redémarrer SSH
sudo systemctl restart ssh
```

::

::callout{icon="i-heroicons-check-circle" color="green"}
#summary
Bonne Pratique

#content
Désactiver le login root SSH force l'utilisation d'utilisateurs nominatifs, améliorant la traçabilité et la sécurité.
::

## Sécurisation SSH

SSH (Secure Shell) est le principal vecteur d'accès à votre serveur. Son durcissement est **critique** pour prévenir les intrusions.

::callout{icon="i-heroicons-exclamation-circle" color="red"}
#summary
🚨 Statistique Importante

#content
Un serveur avec SSH sur le port 22 reçoit des milliers de tentatives de connexion malveillantes par jour. Changer le port réduit drastiquement ce bruit.
::

### Modification du Port SSH

#### Pourquoi Changer le Port par Défaut ?

::callout{icon="i-heroicons-check-circle" color="green"}
#summary
Avantages du Changement de Port

#content

- ✅ Réduction du bruit des scans automatisés (bots)
- ✅ Limitation de la surface d'attaque
- ✅ Logs plus lisibles (moins de tentatives d'intrusion)
- ✅ Conformité avec certaines politiques de sécurité
  ::

::callout{icon="i-heroicons-exclamation-triangle" color="amber"}
⚠️ **Important** : Changer le port SSH n'est **pas une protection suffisante** à elle seule. C'est une couche de défense supplémentaire, pas un remplacement pour une authentification forte (clés SSH) et un firewall.
::

#### Configuration pour Ubuntu 23.04+

Ubuntu 23.04 et versions ultérieures utilisent `systemd socket activation` pour SSH.

::steps

#### Identifier le fichier de configuration

```bash [check-ssh-socket.sh]
# Vérifier si ssh.socket est utilisé
systemctl status ssh.socket

# Localiser le fichier de configuration
ls -la /lib/systemd/system/ssh.socket
```

#### Éditer la configuration socket

```bash [edit-socket.sh]
# Ouvrir le fichier avec nano
sudo nano /lib/systemd/system/ssh.socket
```

#### Modifier le port d'écoute

```ini [/lib/systemd/system/ssh.socket]
[Socket]
# Port personnalisé (choisir entre 49152-65535 pour éviter les conflits)
ListenStream=49152
Accept=no
```

::callout{icon="i-heroicons-light-bulb" color="blue"}
💡 **Choix du port** : Les ports 49152-65535 sont des ports dynamiques/privés rarement utilisés par d'autres services. Évitez les ports communs comme 2222 ou 2022.
::

#### Recharger la configuration systemd

```bash [reload-systemd.sh]
# Recharger les fichiers de configuration systemd
sudo systemctl daemon-reload

# Redémarrer le service SSH
sudo systemctl restart ssh.service

# Vérifier le statut
sudo systemctl status ssh.service
```

#### Vérifier l'écoute du nouveau port

```bash [check-port.sh]
# Afficher les ports en écoute
sudo ss -tlnp | grep ssh

# Ou avec netstat
sudo netstat -tulnp | grep ssh
```

Vous devriez voir :

```
LISTEN    0    128    0.0.0.0:49152    0.0.0.0:*    users:(("sshd",pid=1234,fd=3))
```

#### Configurer le firewall

**Avant** de fermer la session actuelle, configurez le firewall pour autoriser le nouveau port :

```bash [firewall-config.sh]
# Avec UFW (Ubuntu Firewall)
sudo ufw allow 49152/tcp
sudo ufw delete allow 22/tcp
sudo ufw enable

# Vérifier les règles
sudo ufw status numbered
```

::callout{icon="i-heroicons-exclamation-circle" color="red"}
#summary
🚨 CRITIQUE - Risque de Blocage

#content
Ne fermez **jamais** votre session SSH actuelle avant d'avoir vérifié que vous pouvez vous connecter avec le nouveau port dans une **seconde session**. Sinon, vous risquez de vous retrouver bloqué hors de votre serveur.
::

#### Tester la connexion avec le nouveau port

Dans un **nouveau terminal** (sans fermer l'ancien) :

```bash [test-connection.sh]
# Se connecter avec le nouveau port
ssh nom_utilisateur@adresse_ip_serveur -p 49152

# Avec une clé SSH spécifique
ssh -i ~/.ssh/ma_cle nom_utilisateur@adresse_ip_serveur -p 49152
```

#### Finaliser et redémarrer

Une fois la connexion vérifiée sur le nouveau port :

```bash [reboot.sh]
# Redémarrer le serveur pour valider les changements
sudo reboot
```

::

### Configuration SSH Additionnelle

Au-delà du changement de port, d'autres configurations SSH renforcent la sécurité.

#### Éditer sshd_config

```bash [edit-sshd-config.sh]
sudo nano /etc/ssh/sshd_config
```

#### Paramètres de Sécurité Recommandés

```bash [/etc/ssh/sshd_config]
# Désactiver le login root
PermitRootLogin no

# Authentification par clé uniquement (désactiver les mots de passe)
PubkeyAuthentication yes
PasswordAuthentication no
PermitEmptyPasswords no

# Désactiver l'authentification par challenge-response
ChallengeResponseAuthentication no

# Limiter les utilisateurs autorisés
AllowUsers nom_utilisateur autre_utilisateur

# Timeout de connexion
LoginGraceTime 30
ClientAliveInterval 300
ClientAliveCountMax 2

# Désactiver le X11 forwarding si non utilisé
X11Forwarding no

# Protocole SSH version 2 uniquement
Protocol 2

# Limiter les tentatives d'authentification
MaxAuthTries 3
MaxSessions 10
```

::callout{icon="i-heroicons-check-circle" color="green"}
#summary
✅ Authentification par Clé SSH

#content
C'est la méthode la plus sécurisée. Générez une paire de clés avec `ssh-keygen -t ed25519 -C "votre@email.com"` et copiez la clé publique sur le serveur avec `ssh-copy-id`.
::

#### Redémarrer SSH après modifications

```bash [restart-ssh.sh]
# Tester la configuration avant de redémarrer
sudo sshd -t

# Si aucune erreur, redémarrer
sudo systemctl restart ssh
```

## Bonnes Pratiques de Sécurité

::callout{icon="i-heroicons-check-circle" color="green"}
#summary
Best Practices de Sécurité

#content

- ✅ **Mettez à jour régulièrement** : `sudo apt update && sudo apt upgrade` au moins hebdomadairement
- ✅ **Utilisez un firewall** : UFW ou iptables pour contrôler le trafic entrant/sortant
- ✅ **Installez Fail2Ban** : Bannit automatiquement les IP avec trop de tentatives échouées
- ✅ **Activez les mises à jour automatiques** : Avec `unattended-upgrades` pour les patches de sécurité
- ✅ **Surveillez les logs** : `/var/log/auth.log` pour les tentatives de connexion
- ✅ **Sauvegardez régulièrement** : Automatisez les backups avec cron et rsync
- ✅ **Utilisez 2FA pour SSH** : Avec Google Authenticator ou Duo
  ::

### Installation de Fail2Ban

```bash [install-fail2ban.sh]
# Installer Fail2Ban
sudo apt install fail2ban -y

# Copier la configuration par défaut
sudo cp /etc/fail2ban/jail.conf /etc/fail2ban/jail.local

# Éditer la configuration
sudo nano /etc/fail2ban/jail.local
```

Configuration SSH recommandée :

```ini [/etc/fail2ban/jail.local]
[sshd]
enabled = true
port = 49152
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
bantime = 3600
findtime = 600
```

```bash [restart-fail2ban.sh]
# Redémarrer Fail2Ban
sudo systemctl restart fail2ban

# Vérifier le statut
sudo fail2ban-client status sshd
```

## Pièges à Éviter

::callout{icon="i-heroicons-exclamation-triangle" color="red"}
#summary
Common Pitfalls

#content

- ❌ **Ne jamais fermer la session SSH** avant de vérifier le nouveau port dans une seconde session
- ❌ **Ne pas oublier de configurer le firewall** pour le nouveau port SSH
- ❌ **Ne jamais désactiver l'authentification par mot de passe** avant d'avoir testé la connexion par clé SSH
- ❌ **Ne pas utiliser des ports réservés** (< 1024) sans privilèges root appropriés
- ⚠️ **Attention aux règles firewall trop restrictives** qui pourraient vous bloquer l'accès
- ⚠️ **Documenter votre configuration** : Notez le port SSH personnalisé dans un gestionnaire de mots de passe
  ::

## Monitoring et Maintenance

### Surveiller les Connexions SSH

```bash [monitor-ssh.sh]
# Afficher les dernières connexions réussies
last -a

# Afficher les tentatives de connexion échouées
sudo grep "Failed password" /var/log/auth.log | tail -n 20

# Connexions SSH actives
who

# Surveillance en temps réel des logs SSH
sudo tail -f /var/log/auth.log
```

### Mises à Jour Automatiques

```bash [setup-auto-updates.sh]
# Installer unattended-upgrades
sudo apt install unattended-upgrades -y

# Configurer
sudo dpkg-reconfigure unattended-upgrades

# Éditer la configuration
sudo nano /etc/apt/apt.conf.d/50unattended-upgrades
```

Configuration recommandée :

```bash [/etc/apt/apt.conf.d/50unattended-upgrades]
Unattended-Upgrade::Allowed-Origins {
    "${distro_id}:${distro_codename}-security";
};
Unattended-Upgrade::AutoFixInterruptedDpkg "true";
Unattended-Upgrade::Remove-Unused-Dependencies "true";
Unattended-Upgrade::Automatic-Reboot "false";
```

## Checklist Post-Installation

::callout{icon="i-heroicons-check-circle" color="green"}
#summary
Pre-Publication Checklist

#content

- Système mis à jour (`apt update && apt upgrade`)
- Configuration clavier et locales
- Utilisateur sudo créé et testé
- Login root SSH désactivé
- Port SSH modifié et testé
- Authentification par clé SSH configurée
- Authentification par mot de passe SSH désactivée
- Firewall UFW configuré et actif
- Fail2Ban installé et configuré
- Mises à jour automatiques activées
- Sauvegardes configurées
- Monitoring des logs en place
- Configuration documentée dans un endroit sécurisé
  ::

## Conclusion

La sécurisation d'un serveur Ubuntu est un processus continu qui commence dès les premières minutes après le provisionnement. En suivant ce guide, vous établissez une base solide qui protège contre les attaques les plus courantes.

N'oubliez pas que la sécurité est une stratégie en couches : aucune mesure unique n'est infaillible, mais leur combinaison crée une défense robuste. Restez vigilant, surveillez vos logs, et maintenez votre système à jour.
